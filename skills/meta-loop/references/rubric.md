# 元循环评估评分表（模式 A）

每个维度评分为 **通过 / 部分 / 缺口**。第 **2、6、8、9** 维是最常被漏掉的盲点，应重点审查。每一维都回答四件事：问什么、好设计长什么样、缺失时如何失败、方法依据。

> 下面提到的研究数字都来自具体论文设置中的点估计，只能作为定性参考，不应当作硬指标。

## 1. 目标点 / 完成定义

- **问什么**：是否有清晰外部停止目标？它是严重度下限，而不是“直到干净”吗？循环指向的检查是否是对意图的可信分解代理，而不是单个整体判断？
- **好设计**：固定两件事：一是共享严重度词汇中的门禁阈值，例如“没有 X 级及以上阻塞发现”；二是每个产物有独立 oracle：代码用可执行测试，文本用分解评分表。通过 oracle 是必要非充分条件。
- **缺失会怎样失败**：“直到干净”永不终止；整体分数会被冗长和格式刷分；循环优化弱代理，得到看似通过但质量低的产物。
- **依据**：TDD 要求先写失败测试；测试即规格但不完备；G-Eval 表明分解评分优于整体评分。

## 2. 检查者独立性 + judge 去偏（盲点）

- **问什么**：checker 是否和 maker 解耦？是否主动控制 LLM judge 的自我偏好、位置偏差和冗长偏差？
- **好设计**：分两项测试。A：上下文独立，reviewer 在新鲜/隔离上下文运行，使用对抗提示，不延续 maker 推理链。B：高风险时做偏差加固：有真值时对独立参考评审；成对比较交换顺序，不一致降为平局；中和长度；必要时使用不同模型族或小组。残余同模型 rubber-stamp 风险要命名，并用 catch-rate 审计。
- **缺失会怎样失败**：同模型 maker+judge 倾向放大 pass，自我偏好导致橡皮图章；循环看似收敛但质量停滞。
- **依据**：MT-Bench 对位置、冗长、自我增强偏差的观察；自我识别与自我偏好相关；单模型难以纠正自身推理错误。

## 3. 有界迭代 + 明确退出

- **问什么**：是否有硬轮次上限？耗尽后是否升级/交接，而不是静默发货？
- **好设计**：小的固定上限，例如 3 轮。最后一轮修复就是最终动作，不再额外自动加轮。耗尽时把当前结果和残余风险交给人类，绝不宣称成功。
- **缺失会怎样失败**：无界修正烧掉预算，越过质量峰值，或因为 reviewer 总能找小问题而无限循环；成本膨胀后循环被关掉。
- **依据**：Self-Refine 和 Constitutional AI 都显示收益不是单调增加；长 agent 链会成本乘法和错误累积。

## 4. 严重度分级门禁

- **问什么**：checker 是否输出逐条、步骤级、严重度分级发现，而不是一个粗糙 pass/fail？是否只有定义好的阻塞级别进入迭代？
- **好设计**：发现使用下游门禁共享的严重度词汇；阻塞带进入迭代，其他进入非阻塞携带列表。每个阻塞发现都有 maker 处置：复现后修复、有证据拒绝误报、无法判断则默认阻塞。
- **缺失会怎样失败**：二元结论隐藏失败位置；所有问题都阻塞会过度阻塞，真实问题不阻塞会放过；静默丢弃会重新引入缺陷。
- **依据**：过程监督优于只看结果；G-Eval 使用分标准表单。

## 5. 发散 / 回归升级

- **问什么**：循环是否能识别自己卡在同一缺陷上？是否能识别已修好的问题重新出现？
- **好设计**：从第 2 轮开始，每个阻塞发现有签名。若当前签名匹配未解决签名，则升级为卡住；若匹配已解决签名，则升级为回归。两者都不抹掉本轮其他修复。
- **缺失会怎样失败**：循环把预算全花在修不掉的问题上；或 A→B→A 来回摆动，看起来有进展，其实没有收敛。
- **依据**：Reflexion 关于局部最小值的警告；长 agent 循环的回归和错误累积风险。

## 6. 跨运行记忆（盲点）

- **问什么**：循环是否保留经验，包括运行内工作记忆和跨运行组织学习？
- **好设计**：运行内保留未解决签名、已尝试签名、收尾备忘。跨运行记录 catch-rate：便宜机器层漏掉、循环抓到的阻塞发现、轮数和成本，并连接到 kill criterion。可重复问题应固化成永久检查。
- **缺失会怎样失败**：每次都重复同类错误；没人知道循环是否仍值得成本；已修问题没有永久 guard。
- **依据**：Reflexion 的有界 episodic memory；回归测试作为永久 guard。

## 7. 与“完成声明”分离

- **问什么**：循环的权限上限是否明确？它只是提高质量，还是能宣称完成/合并？
- **好设计**：明确自治上限。默认只负责质量：永不宣称完成、永不合并，交给人类/CI。循环通过不等于任务完成。失败应 fail-open，不应 fail-through。
- **缺失会怎样失败**：内部循环自证完成，移除了人类或 CI 责任；内部 pass 被误解为交付完成。
- **依据**：agentic workflow 中从环境获取 ground truth 和控制自治成本的原则。

## 8. 触发可靠性（盲点，最重要）

- **问什么**：循环是否每次都在完成里程碑触发？触发可靠性是否被测量，而不是文字提醒？
- **好设计**：触发绑定到可检测硬动作，如 pre-commit / pre-PR hook，并要求有意识确认。真实会话中审计跳过率。无法 hook 的里程碑要诚实命名，并用文字和审计覆盖。
- **缺失会怎样失败**：循环存在但关键时刻被静默跳过。即使提示写在指令里，也可能因为完成动作发生在几十次工具调用后而被遗忘。
- **依据**：这是设计可靠性论证：完美循环如果不运行，价值为零。

## 9. 成本 / 延迟上限（盲点）

- **问什么**：每轮是否足够便宜，能每次运行？额外 agent/轮次是否比便宜替代方案更值得？
- **好设计**：每轮只跑目标检查，完整重矩阵留到收尾；设置单次 token/时间/成本上限和熔断器，而不是只记录；多 critic 或辩论只用于高风险、难判断场景。
- **缺失会怎样失败**：循环太慢太贵后被习惯性跳过，进而破坏触发可靠性。
- **依据**：CI “保持构建快速”；agent 成本/延迟原则；多 agent 辩论在相同预算下不总优于便宜自修正。

## 10. 可观察性 / 可审计性

- **问什么**：每次运行是否留下可复现、可检查记录？发现了什么、修了什么、拒绝了什么、升级了什么？
- **好设计**：每次运行输出结构化收尾记录，包含核心产物、阻塞修复、有证据拒绝的误报、升级项、非阻塞列表、残余风险、轮数和成本；另有追加式日志记录 catch-rate 和成本。发现和结论基于评分表，可挑战。
- **缺失会怎样失败**：没人知道循环是否抓到问题，无法裁决被拒绝发现，也无法决定保留或删除循环；人类会过度信任黑盒分数。
- **依据**：过程监督的可审计标签；G-Eval 的分标准推理；Constitutional AI 的原则化 critique。

## 示例：评估一个典型 agent 自评循环

被评估循环：agent 在声明代码改动完成前，派一个 reviewer 子 agent。子 agent 使用同模型、新鲜上下文、对抗提示和检查清单，输出严重度分级发现；agent 修复阻塞问题；第 2 轮开始，重复未解决签名会升级；最多 3 轮；写收尾备忘并追加 catch-rate 日志；永不合并。缺点：reviewer 与 maker 是同模型，没有顺序/长度去偏；记录 token 成本但没有预算上限；触发依赖项目指令里的“完成前运行自评”。

| # | 维度 | 结论 | 原因 |
|---|---|---|---|
| 1 | 目标点 | 通过（有保留） | 使用严重度下限，不是直到干净；但清单质量本身不由循环保证 |
| 2 | 独立性 + 去偏 | 部分 | 新鲜上下文和对抗提示通过；同模型且无去偏，仍有自我偏好 |
| 3 | 有界迭代 | 通过 | 3 轮上限，耗尽交给人类 |
| 4 | 严重度门禁 | 通过 | 逐条分级，只迭代阻塞项 |
| 5 | 发散 / 回归 | 部分 | 有未解决重复升级，但无已解决签名追踪，漏 A→B→A |
| 6 | 跨运行记忆 | 部分 | 有运行内记忆和日志，但日志未接 kill criterion |
| 7 | 权限上限 | 通过 | 永不合并，永不自称完成 |
| 8 | 触发可靠性 | 缺口 | 只是 prose pointer，无硬动作、确认和跳过率审计 |
| 9 | 成本上限 | 缺口 | 只记录成本，没有上限 |
| 10 | 可审计性 | 通过 | 有结构化记录和追加日志 |

最重要修复顺序：先修触发可靠性和成本上限；再视风险修 judge 去偏；最后补回归追踪和 kill criterion。
