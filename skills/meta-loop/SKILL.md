# 元循环（Meta-Loop）

这里的 **循环** 指任何有边界的 **产出 → 独立检查 → 修复 → 复查** 过程：验证循环、评审循环、maker/checker 组合、评价器-优化器，或 agent 自我修正循环。本技能做两件事，并以成熟方法论为基础（TDD、Self-Refine、Reflexion、Constitutional AI、LLM-as-judge 研究、过程监督、agentic workflow 模式，见 `references/`）：

- **模式 A：评估** 一个循环是否符合 10 维质量评分表。
- **模式 B：构建** 一个新的循环，按 9 个步骤完成设计。

## 四个盲点（先读这里）

即使能力很强的 agent，在冷启动状态下评估或设计循环时，也会系统性漏掉同样四个维度。该技能的基线测试中，两个独立 agent 各自列出了约 30 个设计维度，却都漏掉了下面四项；而且二者列表互不一致，因为没有共同评分表时，每个 agent 都会临时发明自己的检查清单。这四项正是本技能存在的原因：

- **触发可靠性**：它们会假设循环“会被调用”，却不问它是否会在关键时刻被静默跳过。一个设计再好的循环，如果从不触发，价值就是零。**这是第一盲点。**
- **跨运行记忆 + 终止标准**：它们最多记录单次运行，不会问这个循环长期是否仍然值得付出成本；如果它不再抓到任何问题，就应该缩窄或废弃。它们也很少把经验跨运行带走（Reflexion 风格）。
- **成本会诱导跳过**：它们会说“先跑便宜检查”，却不设置单次运行预算上限，也不会把“过贵 → 被静默关闭 → 触发失败”连起来看。
- **同模型自我偏好**：它们把“派了一个子 agent”当成独立性，却忽略同模型 maker+judge 有自我偏好偏差；也没有顺序/长度去偏。

## 什么时候使用

- 判断一个已经建好或正在提议的循环是否足够好，以及缺什么。
- 设计新的验证、评审、maker-checker 或迭代修正循环。
- 审计或比较循环设计。

## 什么时候不要使用

- 没有“检查并修复迭代”结构的任务：一次性脚本、普通问答。
- 执行一个已经存在的具体循环：那就直接运行该循环。本技能是 **元层**。

## 模式 A：评估循环（10 维评分表）

每一维打分：**通过 / 部分 / 缺口**。**第 2、6、8、9 维是四个盲点，必须重点审查。** 完整标准（好设计长什么样 / 缺失会怎样失败 / 方法依据）和示例见 [references/rubric.md](references/rubric.md)。

| # | 维度 | 一句话：好设计长什么样 |
|---|---|---|
| 1 | 目标点 / 完成定义 | 停止目标是“严重度下限”，不是“直到完全干净”；在产物之前或独立于产物存在 oracle：代码用可执行测试，文本用分解评分表；通过它是必要非充分条件 |
| 2 | 检查者独立性 + judge 去偏 | **A** 新鲜上下文 + 对抗式提示 + 不延续 maker 推理链（二元）；**B** 顺序互换、长度中和、跨模型或小组评审（分级，高风险场景才需要） |
| 3 | 有界迭代 + 明确退出 | 小的硬上限；最后一轮修复就是最终动作；耗尽后升级给人类，绝不“照发不误” |
| 4 | 严重度分级门禁 | 每条发现都有步骤级严重度；只有阻塞级别进入迭代；每条发现都有明确处置，不允许静默丢弃 |
| 5 | 发散 / 回归升级 | 未解决签名重复时升级；同时记录已解决签名，若同一问题重新出现则标记为回归 |
| 6 | 跨运行记忆 | 运行内记忆（未解决签名 + 收尾备忘）加跨运行 catch-rate 日志，并连接到废弃/缩窄标准；可重复问题尽量固化成永久检查 |
| 7 | 与“完成声明”分离 | 明确权限上限，例如只负责质量提升，永不宣称完成、永不合并；“循环通过”不等于“工作完成”；失败应 fail-open，不应 fail-through |
| 8 | 触发可靠性 | 触发绑定到完成里程碑上的可检测硬动作（如 pre-commit / pre-PR hook）和有意识确认；审计跳过率；无法 hook 的里程碑要诚实说明 |
| 9 | 成本 / 延迟上限 | 每轮比收尾门禁更轻；设置明确单次预算上限 / 熔断器，而不只是记录；每增加一轮或一个 critic 都要和更便宜方案比较 |
| 10 | 可观察性 / 可审计性 | 结构化收尾记录 + 追加日志；批评是基于评分表、可挑战的论证，不是黑盒分数；残余风险前置暴露 |

## 模式 B：构建循环（9 步）

完整细节、标准控制骨架、输出 schema、方法依据和开放设计问题见 [references/building-a-loop.md](references/building-a-loop.md)。

1. **先命名唯一核心产物和它的 oracle。** 代码产物用可执行 oracle，并先写失败测试（TDD）。文本产物用在产物之前/独立于产物写好的分解评分表。没有可执行 oracle 不等于不能做循环；固定评分表是更弱但有效的 oracle。
2. **选择 maker/checker 结构并强制独立性。** 默认使用分离的评价器-优化器；检查者使用新鲜上下文和对抗式提示；有真值时对照独立生成的参考；高风险场景再升级到跨模型或小组；加入顺序/长度去偏。
3. **设置目标点和严重度门禁。** 完成是严重度下限，不是“直到干净”；绑定共享严重度词汇；明确阻塞带和携带列表；每条发现必须有处置。
4. **设置迭代边界和发散/回归升级。** 小的硬上限；最后一轮修复就是最终动作；重复未解决签名和已解决后再出现的签名都升级，但不抹掉本轮其他修复。
5. **选择触发器并让它可靠。** 触发绑定到完成里程碑的硬动作和有意识确认；无法 hook 的里程碑要说清；审计跳过率。**不触发的循环毫无价值。**
6. **决定记忆：运行内 vs 跨运行。** 运行内保留未解决签名和工作记忆；跨运行用 catch-rate 日志连接到废弃标准；明确是否需要 Reflexion 式跨产物学习。
7. **设置权限上限。** 默认是只负责质量：永不宣称完成、永不合并、总是交给人类或 CI；完成是单独门禁；fail-open，不 fail-through。
8. **设置成本上限和节奏。** 每轮只跑目标检查，完整重检查留到收尾；设置单次运行预算上限；额外轮次/critic 必须证明比便宜方案更值得。太贵的循环会被跳过。
9. **决定循环如何验证自己。** 定义收尾记录和日志行；先让循环审查自己的设计；记录哪些东西故意不做机器检查；一开始就接入废弃标准。

## 常见错误

- 把“所有机器检查绿了”或“派了一个子 agent”当成“已独立评审”：同模型自我偏好仍然存在。
- 设计了循环却没有保证它会触发：最常见、代价最高。
- 把退出条件设为“直到干净”：它不会收敛。
- 只检测“还在失败”，却漏掉 A→B→A 的回归。
- 只记录成本却不设置上限：循环变贵后会被静默关闭。

## 非目标

- 本技能不替代具体循环的执行；它是元层。
- 本技能不是 CI、cron 或 hook 的实现。
- 评分表是判断脚手架，不是自动门禁：通过 / 部分 / 缺口仍需人类判断。
